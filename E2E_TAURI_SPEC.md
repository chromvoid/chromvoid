# E2E Testing Spec for Tauri Application

## 1. Цель

Определить единый подход к end-to-end (E2E) тестированию Tauri-приложения, чтобы проверять полный пользовательский путь в условиях, максимально приближенных к реальным.

---

## 2. Область покрытия

E2E-тесты должны покрывать:

1. Запуск приложения (cold start)
2. Onboarding / авторизацию (если есть)
3. Ключевые пользовательские сценарии (CRUD, навигация, фильтрация и т.д.)
4. Обработку ошибок (валидация, сетевые сбои, таймауты)
5. Сохранение и восстановление состояния после перезапуска
6. Выход / завершение сессии (если применимо)

Не входят в scope E2E:
- Низкоуровневая логика (unit tests)
- Изолированные backend-команды без UI (integration tests)

---

## 3. Технологический стек

Рекомендуемый стек для Tauri E2E:

- **tauri-driver** (WebDriver endpoint для Tauri)
- **WebdriverIO** (или аналог, например Playwright через WebDriver-bridge)
- **Node.js test runner** (Mocha/Jest/Vitest в зависимости от команды)

Минимальные требования:
- Приложение запускается в test-режиме
- UI-элементы имеют стабильные селекторы (`data-testid`)
- Тесты выполняются локально и в CI

---

## 4. Тестовое окружение

### 4.1 Профиль приложения

Создать отдельный профиль запуска `test`, который:
- использует отдельную директорию данных
- не влияет на пользовательские реальные данные
- может включать флаги для ускорения/детерминизма

### 4.2 Данные

- Каждый тест должен быть изолированным
- Перед тестом: подготовка данных (`seed`)
- После теста: очистка данных (`teardown`)

### 4.3 Сеть и внешние API

- Внешние API по возможности мокать
- Тесты не должны зависеть от нестабильной сети
- Для критичных интеграций выделить отдельные smoke-тесты

---

## 5. Структура сценариев

Каждый E2E сценарий оформляется как:

1. **Given**: начальное состояние
2. **When**: действия пользователя
3. **Then**: ожидаемый результат

Пример:
- Given: пользователь на главном экране
- When: создает новую задачу
- Then: задача отображается в списке и сохраняется после перезапуска

---

## 6. Обязательные E2E сценарии (MVP)

### E2E-001: Первый запуск
- Приложение открывается без падений
- Отображается стартовый экран

### E2E-002: Авторизация (если есть)
- Успешный вход с валидными данными
- Ошибка при невалидных данных

### E2E-003: Основной пользовательский флоу
- Создание сущности
- Редактирование сущности
- Удаление сущности
- Проверка состояния в UI

### E2E-004: Перезапуск
- После перезапуска данные/сессия восстановлены корректно

### E2E-005: Обработка ошибок
- Симуляция сетевой ошибки
- Корректный текст ошибки
- Возможность повторить действие

---

## 7. Селекторы и стабильность тестов

### 7.1 Правила селекторов

Использовать в первую очередь:
- `data-testid="..."`

Избегать:
- CSS-селекторов, завязанных на визуальную верстку
- Текстов, подверженных частым изменениям

### 7.2 Ожидания

- Использовать явные ожидания (`waitForDisplayed`, `waitUntil`)
- Не использовать `sleep` без необходимости

---

## 8. Критерии качества E2E

1. **Детерминизм**: один и тот же результат при повторном запуске
2. **Надежность**: flaky rate < 2%
3. **Скорость**: smoke suite ≤ 10 минут в CI
4. **Наблюдаемость**: логи, скриншоты и артефакты при падении

---

## 9. CI/CD интеграция

В CI необходимо:

1. Поднять окружение
2. Собрать/запустить Tauri-приложение в test-режиме
3. Запустить tauri-driver
4. Выполнить E2E suite
5. Сохранить артефакты:
   - скриншоты
   - видео (опционально)
   - логи тест-раннера

Рекомендация:
- Запускать smoke-набор на каждый PR
- Полный regression-набор по расписанию/перед релизом

---

## 10. Репортинг и triage

При падении теста фиксировать:

- ID сценария (например, E2E-003)
- шаг и ожидаемый результат
- фактический результат
- приложенные артефакты
- версию приложения и ОС

При повторяющихся flaky-падениях:
- маркировать тест как flaky
- заводить техдолг-задачу на стабилизацию

---

## 11. Definition of Done для E2E покрытия фичи

Фича считается покрытой E2E, если:

- добавлен минимум 1 happy-path тест
- добавлен минимум 1 негативный тест (ошибка/валидация)
- тесты проходят локально и в CI
- обновлена документация (этот spec или связанный раздел)

---

## 12. Версионирование spec

- Любые изменения в пользовательских сценариях должны сопровождаться обновлением этого файла
- Изменения фиксируются в PR вместе с обновлением тестов

---

## 13. Быстрый checklist перед релизом

- [ ] Smoke E2E пройден
- [ ] Регресс E2E пройден (или зафиксированы исключения)
- [ ] Нет критических flaky тестов
- [ ] Ключевые сценарии подтверждены на целевых ОС
- [ ] Артефакты тестового прогона сохранены

---


2. Внедрить naming convention (`E2E-XXX`)
3. Добавить pre-merge smoke запуск
4. Ввести еженедельный review flaky-тестов
